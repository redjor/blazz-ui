## Overview

The Badge component is a compact visual indicator used to highlight status, categorize content, display counts, or draw attention to specific information. Built with `class-variance-authority`, it provides multiple variants to communicate different types of information effectively. Badges are non-interactive by default but can be made interactive when needed.

Key features:

- **Multiple variants**: 6 visual styles (default, secondary, destructive, outline, ghost, link)
- **Flexible content**: Supports text, numbers, icons, or combinations
- **Compact design**: Small footprint perfect for inline use
- **Semantic colors**: Color-coded variants communicate meaning
- **Icon support**: Built-in icon styling with proper sizing and spacing
- **TypeScript**: Fully typed component with IntelliSense support

## Usage Guidelines

### When to Use

- **Status indicators**: Show the state of items (active, pending, completed, failed)
- **Categories and tags**: Categorize content with labels
- **Counts and notifications**: Display numeric values like unread counts or quantities
- **Metadata**: Show additional information like dates, versions, or types
- **User roles**: Identify user permissions or account types
- **Product features**: Highlight "New", "Sale", "Featured", or other attributes
- **Priority levels**: Indicate urgency or importance

### When Not to Use

- **Primary actions**: Use Button component for clickable actions
- **Long text content**: Badges are for short labels only
- **Form validation**: Use proper error messages with Input components
- **Navigation**: Use appropriate navigation components
- **Large amounts of data**: Consider using chips or cards for more complex information

### Best Practices

1. **Keep text short**: Use 1-3 words maximum for clarity
2. **Choose meaningful variants**: Use variant colors that match the semantic meaning (destructive for errors, default for primary status)
3. **Be consistent**: Use the same variant for the same type of information across your app
4. **Don't overuse**: Too many badges create visual clutter
5. **Combine with icons**: Icons enhance understanding, especially for status indicators
6. **Consider context**: Badges should complement, not replace, surrounding context
7. **Maintain readability**: Ensure sufficient contrast between badge and background
8. **Use for supplementary info**: Badges should support primary content, not replace it

## Accessibility

### Keyboard Support

Badges are non-interactive by default and don't receive keyboard focus. If you make badges interactive (clickable), they should:

| Key     | Action              |
| ------- | ------------------- |
| `Tab`   | Move focus to badge |
| `Enter` | Activate badge      |
| `Space` | Activate badge      |

### ARIA Attributes

For non-interactive badges:
- No special ARIA attributes needed
- Screen readers announce badge text as part of surrounding content

For interactive badges:
- **`role="button"`**: If badge is clickable
- **`aria-label`**: If badge text doesn't fully describe the action
- **`aria-pressed`**: For toggle-style badges

For status indicators:
- **`aria-live="polite"`**: For badges that update dynamically
- **`role="status"`**: For status badges that change

### Screen Reader Behavior

- Screen readers announce badge text in reading order
- Icon-only badges should have `aria-label` for proper announcement
- Status badges with icons should have descriptive text (visible or via `aria-label`)
- Numeric badges are announced as numbers ("three" not "three badge")

### Color and Contrast

- All badge variants meet WCAG 2.1 Level AA contrast requirements
- Don't rely solely on color to convey information
- Use icons or text to supplement color-based meaning
- Test in dark mode and with color blindness simulators

## Technical Details

### Component API

Based on `@base-ui/react` with `useRender` for flexible rendering options.

### Styling

Uses `class-variance-authority` (CVA) for variant management:

**Variants**:

- `default`: Primary blue background with white text
- `secondary`: Secondary gray background
- `destructive`: Red background for errors or warnings (reduced opacity for subtlety)
- `outline`: Transparent with border
- `ghost`: Transparent background with hover state
- `link`: Styled as a text link

**Base styles**:
- Height: `h-5` (1.25rem)
- Padding: `px-2 py-0.5`
- Typography: `text-xs font-medium`
- Border radius: `rounded-4xl` (fully rounded)
- Icon size: `size-3` (0.75rem)

**Special features**:
- Icons automatically adjust padding (`has-data-[icon]` selectors)
- SVG icons get proper sizing via `[&>svg]:size-3`
- Focus states for interactive badges
- Transition effects for smooth interactions

### Dependencies

- `@base-ui/react` - Headless UI utilities (`useRender`, `mergeProps`)
- `lucide-react` - Icon library
- `class-variance-authority` - Variant management
- `tailwind-merge` - Class name utility

### Type Definition

```typescript
interface BadgeProps extends useRender.ComponentProps<"span">, VariantProps<typeof badgeVariants> {
	variant?: "default" | "secondary" | "destructive" | "outline" | "ghost" | "link"
	children?: React.ReactNode
	className?: string
	render?: useRender.ComponentProps<"span">["render"]
}
```

### Common Patterns

**Status badge**:
```tsx
<Badge variant="default">
  <Check />
  Active
</Badge>
```

**Notification count**:
```tsx
<Badge variant="destructive">12</Badge>
```

**Tag/category**:
```tsx
<Badge variant="secondary">Electronics</Badge>
```

**Interactive badge (as button)**:
```tsx
<Badge
  variant="outline"
  role="button"
  onClick={() => handleClick()}
  className="cursor-pointer"
>
  Filter
  <X />
</Badge>
```

**With custom styling**:
```tsx
<Badge variant="default" className="uppercase tracking-wider">
  Premium
</Badge>
```
