## Overview

The Button component is a fundamental interactive element built on top of `@base-ui/react`. It provides a consistent, accessible, and customizable way to trigger actions throughout your application. With support for multiple variants, sizes, icons, and states, the Button component can adapt to any design need while maintaining accessibility and usability standards.

Key features:

- **Multiple variants**: 6 visual styles (default, outline, secondary, ghost, destructive, link)
- **Flexible sizing**: 8 size options including icon-only variants
- **Icon support**: Built-in support for Lucide icons positioned before or after text
- **Accessible**: Full keyboard navigation and ARIA attribute support
- **TypeScript**: Fully typed with IntelliSense support

## Usage Guidelines

### When to Use

- **Primary actions**: Use the default variant for the primary action on a page (e.g., "Save", "Submit", "Create")
- **Secondary actions**: Use outline or secondary variants for less prominent actions
- **Navigation**: Use ghost or link variants for navigation actions
- **Destructive actions**: Use the destructive variant for delete or destructive operations
- **Inline actions**: Use ghost or link variants for actions within text content

### When Not to Use

- **Navigation links**: Use Next.js Link component wrapped in a button style if you need routing
- **Form submissions without validation**: Always validate forms before submission
- **Non-interactive elements**: Don't use buttons for display-only elements; use appropriate semantic HTML
- **Too many primary buttons**: Limit to one primary button per section to maintain clear hierarchy

### Best Practices

1. **Use clear, action-oriented labels**: Button text should clearly describe the action (e.g., "Save Changes" not "OK")
2. **Maintain visual hierarchy**: Use different variants to establish importance (primary vs. secondary actions)
3. **Provide feedback**: Ensure buttons have visible hover, focus, and active states
4. **Consider icon meaning**: Only use icons that are universally understood or provide clear context
5. **Respect disabled state**: Only disable buttons when an action is temporarily unavailable, not as a permanent state
6. **Loading states**: For async actions, consider adding loading indicators to show progress
7. **Destructive actions**: Always use the destructive variant for delete operations and consider confirmation dialogs

## Accessibility

### Keyboard Support

| Key     | Action                               |
| ------- | ------------------------------------ |
| `Tab`   | Move focus to/from button            |
| `Space` | Activate button                      |
| `Enter` | Activate button (in forms)           |
| `Esc`   | Remove focus (if focused externally) |

### ARIA Attributes

- **`role="button"`**: Automatically applied by the Button primitive
- **`aria-disabled`**: Applied when `disabled` prop is true
- **`aria-expanded`**: Can be set for buttons that control expandable content (e.g., dropdowns)
- **`aria-pressed`**: Can be set for toggle buttons to indicate pressed state
- **`aria-label`**: Should be provided for icon-only buttons to describe the action

### Screen Reader Behavior

- Screen readers announce the button label and role ("Button, [label]")
- When disabled, screen readers announce "disabled" or "unavailable"
- When `aria-expanded` is set, screen readers announce "expanded" or "collapsed"
- Icon-only buttons should always have an `aria-label` for proper screen reader support

### Focus Management

- Buttons receive visible focus indicators that meet WCAG 2.1 contrast requirements
- Focus states use the `focus-visible:ring` utility for keyboard-only focus indicators
- Disabled buttons do not receive focus

## Technical Details

### Component API

Based on `@base-ui/react/button` primitive which provides:

- Proper semantic HTML (`<button>` element)
- Built-in keyboard event handling
- Automatic ARIA attributes
- TypeScript types and IntelliSense support

### Styling

Uses `class-variance-authority` (CVA) for variant management with the following configuration:

**Variants**:

- `default`: Primary blue background with white text
- `outline`: Transparent with border and subtle hover state
- `secondary`: Secondary color background
- `ghost`: Transparent background with hover state
- `destructive`: Red background for destructive actions
- `link`: Styled as a text link with underline on hover

**Sizes**:

- `xs`: Extra small (h-6, text-xs)
- `sm`: Small (h-7, text-sm)
- `default`: Default size (h-8, text-sm)
- `lg`: Large (h-9)
- `icon`: Square icon button (8x8)
- `icon-xs`: Extra small icon button (6x6)
- `icon-sm`: Small icon button (7x7)
- `icon-lg`: Large icon button (9x9)

### Dependencies

- `@base-ui/react` - Headless UI button primitive
- `lucide-react` - Icon library
- `class-variance-authority` - Variant management
- `tailwind-merge` - Class name utility

### Type Definition

```typescript
interface ButtonProps extends ButtonPrimitive.Props, VariantProps<typeof buttonVariants> {
	variant?: "default" | "outline" | "secondary" | "ghost" | "destructive" | "link"
	size?:
		| "default"
		| "xs"
		| "sm"
		| "lg"
		| "icon"
		| "icon-xs"
		| "icon-sm"
		| "icon-lg"
	disabled?: boolean
	children?: React.ReactNode
	className?: string
}
```
