## Overview

The Input component is a fundamental form element that allows users to enter and edit text. Built on `@base-ui/react`, it provides a consistent and accessible interface for data collection. The component supports all standard HTML input types and features visual feedback for different states including focus, disabled, and validation errors.

Key features:

- **Multiple input types**: Support for text, email, password, number, date, file, and more
- **Visual states**: Clear feedback for focus, disabled, and invalid states
- **Icon integration**: Easy integration with icons for enhanced UX
- **Accessible**: Full keyboard navigation and ARIA support
- **Form-ready**: Works seamlessly with HTML forms and validation
- **TypeScript**: Fully typed for better developer experience

## Usage Guidelines

### When to Use

- **Text collection**: For collecting single-line text input from users
- **Forms**: As part of forms for user registration, login, settings, etc.
- **Search**: For search functionality with appropriate type and icons
- **Data entry**: For entering structured data like email, phone, dates
- **File uploads**: For allowing users to select files from their device

### When Not to Use

- **Multi-line text**: Use Textarea component for longer text input
- **Selection from options**: Use Select or Radio Group for predefined choices
- **Boolean values**: Use Checkbox or Switch for true/false values
- **Rich text**: Use a rich text editor for formatted content
- **Read-only data**: Use plain text or display components for non-editable data

### Best Practices

1. **Always use labels**: Pair inputs with Label components for accessibility and clarity
2. **Provide helpful placeholders**: Use placeholder text to show format examples, not as a replacement for labels
3. **Show validation feedback**: Provide clear error messages when validation fails
4. **Choose appropriate types**: Use specific input types (email, tel, url) to enable better keyboard layouts on mobile
5. **Make required fields obvious**: Mark required fields clearly and consistently
6. **Consider auto-complete**: Use appropriate autocomplete attributes for better UX
7. **Disable sparingly**: Only disable inputs when truly necessary, and explain why they're disabled
8. **Group related inputs**: Use logical grouping for related fields in forms

## Accessibility

### Keyboard Support

| Key                  | Action                                   |
| -------------------- | ---------------------------------------- |
| `Tab`                | Move focus to/from input                 |
| `Shift + Tab`        | Move focus to previous element           |
| `Enter`              | Submit form (when in form context)       |
| `Esc`                | Clear input (for search type)            |
| Arrow keys           | Navigate through text                    |
| `Ctrl/Cmd + A`       | Select all text                          |
| `Ctrl/Cmd + C/V/X`   | Copy/paste/cut text                      |

### ARIA Attributes

- **`aria-label`**: Provides accessible label when visual label isn't present
- **`aria-labelledby`**: References the associated label element
- **`aria-describedby`**: References help text or error messages
- **`aria-invalid`**: Indicates validation state (true when input has errors)
- **`aria-required`**: Indicates required fields
- **`aria-disabled`**: Indicates disabled state (prefer using disabled attribute)

### Screen Reader Behavior

- Screen readers announce the input type, label, and current value
- When focused, screen readers announce placeholder text if no value is present
- Invalid inputs are announced as "invalid" along with error messages if `aria-describedby` is set
- Required fields are announced as "required"
- File inputs announce "Choose file" or similar based on browser

### Focus Management

- Inputs receive visible focus indicators that meet WCAG 2.1 contrast requirements
- Focus ring uses the `focus-visible:ring` utility for clear keyboard focus indication
- Focus state is preserved when navigating with keyboard
- Disabled inputs do not receive focus

## Technical Details

### Component API

Based on `@base-ui/react/input` primitive which provides:

- Proper semantic HTML (`<input>` element)
- Built-in keyboard event handling
- Automatic ARIA attributes
- Form integration support
- TypeScript types

### Styling

The Input component uses Tailwind CSS utility classes with conditional states:

**Base styles**:
- Height: `h-8` (2rem)
- Border: `rounded-lg border border-input`
- Padding: `px-2.5 py-1`
- Typography: `text-base` on desktop, `text-sm` on mobile

**States**:
- **Focus**: Blue ring with `focus-visible:ring-ring/50`
- **Invalid**: Red border and ring with `aria-invalid:border-destructive`
- **Disabled**: Reduced opacity and pointer events disabled
- **Dark mode**: Adjusted background and border colors

**Special handling**:
- File input buttons are styled with `file:` modifiers
- Placeholder text uses `placeholder:text-muted-foreground`

### Dependencies

- `@base-ui/react` - Headless UI input primitive
- `tailwind-merge` - Class name utility

### Type Definition


### Common Patterns

**With icon prefix**:
```tsx
<div className="relative">
  <Icon className="absolute left-2.5 top-2 h-4 w-4 text-muted-foreground" />
  <Input placeholder="Search..." className="pl-9" />
</div>
```

**With validation**:
```tsx
<div>
  <Label htmlFor="email">Email</Label>
  <Input
    id="email"
    type="email"
    aria-invalid={hasError}
    aria-describedby="email-error"
  />
  {hasError && (
    <p id="email-error" className="text-xs text-destructive mt-1">
      Invalid email address
    </p>
  )}
</div>
```
